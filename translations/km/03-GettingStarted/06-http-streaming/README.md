# HTTPS ផ្ទេររូបភាពជាមួយ Model Context Protocol (MCP)

ជំពូកនេះផ្តល់ជូនមគ្គុទេសក៍ពេញលេញសម្រាប់ការអនុវត្តផ្ទេររូបភាពដែលមានសុវត្ថិភាព អាចផ្ទុកបានយ៉ាងធំ និងពេលវេលាចៀសវាងជាមួយ Model Context Protocol (MCP) ដោយប្រើ HTTPS។ វាអ្នកគ្របដណ្តប់ពីមូលហេតុនៃការផ្ទេររូបភាព វិធីដឹកជញ្ជូនដែលមានស្រាប់ វិធីអនុវត្ត HTTP អាចផ្ទេររូបភាពក្នុង MCP ហើយវិធីសុវត្ថិភាពល្អបំផុត ការធ្វើបម្រែបម្រួលពី SSE និងមគ្គុទេសក៍ប្រតិបត្តិការសម្រាប់ស្ថាបនាកម្មវិធីប្រើប្រាស់ MCP ផ្ទេររូបភាពរបស់អ្នក។

## វិធីដឹកជញ្ជូន និងការផ្ទេររូបភាពក្នុង MCP

ប្លុកនេះពិនិត្យពីវិធីដឹកជញ្ជូនផ្សេងៗដែលមានប្រើនៅក្នុង MCP និងតួនាទីរបស់វា ក្នុងការអនុញ្ញាតឱ្យមានសមត្ថភាពផ្ទេររូបភាពសម្រាប់ការទំនាក់ទំនងពេលវេលាចៀសវាងរវាងអ្នកដំណើរការនិងម៉ាស៊ីនបម្រើ។

### វិធីដឹកជញ្ជូនមានអ្វីខ្លះ?

វិធីដឹកជញ្ជូនកំណត់តើធាតុទិន្នន័យត្រូវប្ដូរពីម៉ាស៊ីនភ្ញៀវទៅម៉ាស៊ីនបម្រើយ៉ាងដូចម្តេច។ MCP គាំទ្រវិធីដឹកជញ្ជូនច្រើនប្រភេទ ដើម្បីសម្របសម្រួលទៅបរិយាកាសនានា និងតម្រូវការផ្សេងៗ៖

- **stdio**៖ ប្រាក់ចូល-ចេញស្តង់ដា ដែលសមស្របសម្រាប់ឧបករណ៍ក្នុងតំបន់ និង CLI ។ ងាយស្រួល ប៉ុន្តែមិនសមស្របសម្រាប់បណ្តាញឬពពក។
- **SSE (Server-Sent Events)**៖ អនុញ្ញាតម៉ាស៊ីនបម្រើផ្ញើការអាប់ដេត​ពេលវេលាចៀសវាងទៅម៉ាស៊ីនភ្ញៀវតាម HTTP។ ល្អសម្រាប់ UI វេប ប៉ុន្តែនៅតែមានកម្រិតនៅការនឹងផ្ទុក និងភាពត្រឹមត្រូវ។
- **Streamable HTTP**៖ វិធីដឹកជញ្ជូនផ្ទេររូបភាពដែលផ្អែកលើ HTTP ទំនើប ដែលគាំទ្រជូនដំណឹង និងភារកិច្ចផ្ទុកបានល្អ។ ត្រូវបានណែនាំសម្រាប់ស្ថានភាពផ្សេងៗទូទៅនិងពពក។

### តារាងប្រៀបធៀប

សូមមើលតារាងប្រៀបធៀបខាងក្រោម សម្រាប់យល់ពីភាពខុសគ្នានៃវិធីដឹកជញ្ជូនទាំងនេះ ៖

| វិធីដឹកជញ្ជូន         | អាប់ដេតពេលវេលាចៀសវាង  | ផ្ទេររូបភាព     | ភាពអាចផ្ទុកបាន    | ករណីប្រើ              |
|-------------------------|-----------------------------|-------------------|---------------------|------------------------|
| stdio                   | មិន                    | មិន                 | ទាប                 | ឧបករណ៍ CLI ក្នុងតំបន់ |
| SSE                     | មាន                      | មាន                 | មធ្យម                | វេប អាប់ដេតពេលវេលាចៀសវាង  |
| Streamable HTTP         | មាន                      | មាន                 | ខ្ពស់                | ពពក អតិថិជនច្រើន       |

> **គន្លឹះ៖** ការជ្រើសរើសវិធីដឹកជញ្ជូនត្រឹមត្រូវប៉ះពាល់ដល់សមត្ថភាព បរិមាណការផ្ទុក និងបទពិសោធន៍អ្នកប្រើ។ **Streamable HTTP** ត្រូវបានណែនាំសម្រាប់កម្មវិធីទូទៅដែលអាចផ្ទុកបាន និងរួចជាពពក។

ចំណាំពីវិធីដឹកជញ្ជូន stdio និង SSE ដែលបានបង្ហាញក្នុងជំពូកមុន និងរបៀបដែល Streamable HTTP ជាវិធីដឹកជញ្ជូនដែលបានពិភាក្សានៅជំពូកនេះ។

## ការផ្ទេររូបភាព៖ គំនិត និងមូលហេតុចម្បង

ការយល់ដឹងចំពោះគំនិតមូលដ្ឋាន និងមូលហេតុចម្បងចំពោះការផ្ទេររូបភាព គឺមានសារៈសំខាន់សម្រាប់ការអនុវត្តប្រព័ន្ធទំនាក់ទំនងពេលវេលាចៀសវាងមានប្រសិទ្ធភាព។

**ការផ្ទេររូបភាព** គឺជាបច្ចេកទេសក្នុងកម្មវិធីបណ្ដាញ ដែលអនុញ្ញាតឲ្យផ្ញើ និងទទួលទិន្នន័យជាលេខតូចៗ ឬជាជួរតំណរនៃព្រឹត្តិការណ៍ ជំនួសការរំពឹងទុកចំពោះចម្លើយពេញលេញ។ នេះមានប្រយោជន៍បំផុតសម្រាប់៖

- ឯកសារធំនៅឬឯកសារទិន្នន័យធំ។
- ការអាប់ដេតពេលវេលាចៀសវាង (ឧ. ការជជែក, ឬរបារគ្រោងការរីកចម្រើន)។
- ការគណនាដំណើរការដែលមានរយៈពេលវែងដែលអ្នកចង់ឲ្យអ្នកប្រើទទួលបានព័ត៌មានរាល់ពេល។

នេះជាចំណុចដែលអ្នកគួរឡើងដឹងអំពីការផ្ទេររូបភាពនៅកម្រិតខ្ពស់៖

- ទិន្នន័យត្រូវបានផ្ដល់ជាបន្តបន្ទាប់ មិនមែនផ្ដល់ទាំងមូលក្នុងពេលតែមួយទេ។
- អ្នកភ្ញៀវអាចដំណើរការទិន្នន័យបានពេលវាអាចមកដល់។
- ការពន្យារពេលដែលគេដឹងស្រួលធ្លាក់កាត់ និងបង្កើនបទពិសោធន៍អ្នកប្រើ។

### ហេតុអ្វីបានជាយើងប្រើការផ្ទេររូបភាព?

ហេតុផលសំខាន់វាដូចខាងក្រោម៖

- អ្នកប្រើទទួលបានមតិយោបល់ភ្លាមៗ មិនមែននៅចុងបញ្ចប់តែម្តងទេ
- អនុញ្ញាតកម្មវិធីពេលវេលាចៀសវាង និង UI តបតាមភាពប្រសើរឡើង
- ប្រើប្រាស់ធនធានបណ្ដាញ និងគណនា​បានប្រសើរឡើង

### ឧទាហរណ៍ស្រាល៖ ម៉ាស៊ីនបម្រើ និង ម៉ាស៊ីនភ្ញៀវ HTTP ផ្ទេររូបភាព

នេះជាឧទាហរណ៍សាមញ្ញពីរបៀបទទួលបានការផ្ទេររូបភាព៖

#### Python

**ម៉ាស៊ីនបម្រើ (Python, ប្រើ FastAPI និង StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ម៉ាស៊ីនភ្ញៀវ (Python, ប្រើ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ឧទាហរណ៍នេះបង្ហាញពីម៉ាស៊ីនបម្រើផ្ញើសារជាជួរដល់ម៉ាស៊ីនភ្ញៀវពេលវាបាន។ មិនរង់ចាំសារទាំងអស់ដើម្បីបានរួចរាល់ឡើយ។

**របៀបដំណើរការ៖**

- ម៉ាស៊ីនបម្រើបញ្ចេញសារនីមួយៗពេលវាជាស្រេច។
- ម៉ាស៊ីនភ្ញៀវទទួល និងបញ្ជូនសារទៅនៅមុនពេលវាចប់។

**ច្បាប់កំណត់៖**

- ម៉ាស៊ីនបម្រើត្រូវប្រើករណីស្ត្រីមមិចបន្សំព្រឹត្តិការណ៍ (ឧ. `StreamingResponse` ក្នុង FastAPI)។
- ម៉ាស៊ីនភ្ញៀវត្រូវដំណើរការជាលីនុយបន្ទាត់ក្នុងស្ទ្រីម (`stream=True` ក្នុង requests)។
- ប្រភេទបច្ចេកទេសគឺជា `text/event-stream` ឬ `application/octet-stream` ជាធម្មតា។

#### Java

**ម៉ាស៊ីនបម្រើ (Java, ប្រើ Spring Boot និង Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ម៉ាស៊ីនភ្ញៀវ (Java, ប្រើ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ចំណាំអំពីអនុវត្ត Java៖**

- ប្រើ Spring Boot រីករាយជាមួយ `Flux` សម្រាប់ការផ្ទេររូបភាព
- `ServerSentEvent` ផ្ដល់ប្រាប់ព្រឹត្តិការណ៍ស្ទ្រីមជា​ដែលមានប្រភេទព្រឹត្តិការណ៍
- `WebClient` ជាមួយ `bodyToFlux()` អនុញ្ញាតដំណើរការផ្ទេររូបភាពក្រោមវេរុសកម្ម
- `delayElements()` អនុញ្ញាតទាញយកពេលវេលាបញ្ចៀសរវាងព្រឹត្តិការណ៍
- ព្រឹត្តិការណ៍អាចមានប្រភេទ (`info`, `result`) សម្រាប់ការគ្រប់គ្រងល្អរបស់ម៉ាស៊ីនភ្ញៀវ

| លក្ខណៈ                 | ផ្ទេររូបភាព HTTP បែបចាស់      | MCP ផ្ទេររូបភាព (ជូនដំណឹង)   |
|-------------------------|---------------------------------|---------------------------------|
| ចម្លើយសំខាន់            | បំលែងជាផ្នែក                    | តែមួយ នៅចុងក្រោយ               |
| ការអាប់ដេតជំហាន        | ផ្ញើជាផ្នែកទិន្នន័យ             | ផ្ញើជាជូនដំណឹង               |
| តម្រូវការម៉ាស៊ីនភ្ញៀវ   | ត្រូវដំណើរការស្ទ្រីម              | ត្រូវអនុវត្តអ្នកដំណើរការសារ      |
| ករណីប្រើ               | ឯកសារធំ ចំណុច AI ត្រៀមនៃតួ | ជំហាន កំណត់ហេតុ ការវាយតម្លៃពេលវេលាចៀសវាង  |

### ភាពខុសគ្នាសំខាន់ៗ

បន្ថែមពីនេះ មានភាពខុសគ្នាសំខាន់ៗ៖

- **រចនាសម្ព័ន្ធការទំនាក់ទំនង៖**
  - បែបចាស់៖ ប្រើកូដបំលែងផ្នែកសាមញ្ញដើម្បីផ្ញើទិន្នន័យជាផ្នែក
  - MCP៖ ប្រើរបៀបជូនដំណឹងដែលមានរចនាសម្ព័ន្ធជាមួយប្រព័ន្ធ JSON-RPC
  
- **ទ្រង់ទ្រាយសារ៖**
  - បែបចាស់៖ ឯកសារសរសេរទ្រង់ទ្រាយធម្មតាជាមួយបន្ទាត់ថ្មីៗ
  - MCP៖ វត្ថុ LoggingMessageNotification មានរចនាសម្ព័ន្ធ និងមាតាដាតា
  
- **ការអនុវត្តម៉ាស៊ីនភ្ញៀវ៖**
  - បែបចាស់៖ ម៉ាស៊ីនភ្ញៀវសាមញ្ញដំណើរការឆ្លើយតបស្ទ្រីម
  - MCP៖ ម៉ាស៊ីនភ្ញៀវមានភាពស្មុគស្មាញជាមួយអ្នកដំណើរការសារសម្រាប់ដំណើរការប្រភេទសារផ្សេងៗ
  
- **ការអាប់ដេតជំហាន៖**
  - បែបចាស់៖ ជំហានជាជំពូកម្ខាងក្នុងចម្លើយសំខាន់
  - MCP៖ ជំហានត្រូវផ្ញើជាសារជូនដំណឹងផ្សេងទៀត ខណៈដែលចម្លើយសំខាន់ផ្ញើនៅចុងក្រោយ

### ការណែនាំ

មានអ្វីដែលយើងណែនាំពេលជ្រើសរើសអនុវត្តផ្ទេររូបភាពបែបចាស់ (ជាកំណត់ហេតុដែលយើងបានបង្ហាញខាងលើដោយប្រើ `/stream`) ឬជ្រើសរើសផ្ទេររូបភាពតាម MCP។

- **សម្រាប់តម្រូវការផ្ទេរដែលសាមញ្ញ៖** ផ្ទេររូបភាព HTTP បែបចាស់គឺងាយធ្វើ និងគ្រប់គ្រាន់សម្រាប់តម្រូវការ​ផ្ទេររូបភាពមូលដ្ឋាន។
- **សម្រាប់កម្មវិធីស៊ីខ្លាំង និងមានប្រតិកម្ម៖** MCP ផ្ទេររូបភាពផ្ដល់នូវរបៀបរចនាសម្ព័ន្ធជាងល្អមានមាតាដាតាពហុ និងបំបែករវាងជូនដំណឹង និងលទ្ធផលចុងក្រោយ។
- **សម្រាប់កម្មវិធី AI៖** ប្រព័ន្ធជូនដំណឹងរបស់ MCP មានប្រយោជន៍ពិសេសសម្រាប់ការងារពេលវេលាយូរឆ្នាំដែលអ្នកចង់ឲ្យអ្នកប្រើបានទទួលព័ត៌មានអំពីជំហានកំពុងដំណើរការ។

## ការផ្ទេររូបភាពក្នុង MCP

ល្អហើយ អ្នកបានឃើញការណែនាំ និងការប្រៀបធៀបរហូតមកដល់ពេលនេះជាផ្នែកមួយនៃភាពខុសគ្នារវាងផ្ទេរបែបទំនើប និង MCP។ ឥឡូវយើងចូលទៅបរិយាយលម្អិតពីរបៀបដែលអ្នកអាចយកអត្ថប្រយោជន៍ពីការផ្ទេររូបភាពក្នុង MCP។

ការយល់ដឹងពីរបៀបដំណើរការការផ្ទេររូបភាពនៅក្នុងបរិបទ MCP គឺសំខាន់សម្រាប់បង្កើតកម្មវិធីដែលមានប្រតិកម្ម និងផ្ដល់មតិយោបល់ពេលវេលាចៀសវាងដល់អ្នកប្រើនៅពេលអនុវត្តរាល់ប្រតិបត្តិការដ៏វែង។

ក្នុង MCP ការផ្ទេររូបភាពមិនមែនផ្ញើចម្លើយសំខាន់ជាច្រើនផ្នែកទេ ប៉ុន្តែផ្ញើ **ការជូនដំណឹង** ទៅម៉ាស៊ីនភ្ញៀវ ខណៈឧបករណ៍កំពុងដំណើរការសំណើរ។ ការជូនដំណឹងទាំងនេះអាចមានទាំងការអាប់ដេតជំហាន កំណត់ហេតុ ឬព្រឹត្តិការណ៍ផ្សេងៗ។

### របៀបដំណើរការ

លទ្ធផលសំខាន់ត្រូវបានផ្ញើជាចម្លើយតែមួយ។ ទោះយ៉ាងណា ការជូនដំណឹងអាចផ្ញើជាសារផ្សេងៗ ក្នុងពេលកំពុងដំណើរការ ហើយធ្វើអោយម៉ាស៊ីនភ្ញៀវបានអាប់ដេតពេលវេលាចៀសវាង។ ម៉ាស៊ីនភ្ញៀវត្រូវអាចគ្រប់គ្រង និងបង្ហាញការជូនដំណឹងទាំងនេះ។

## ការជូនដំណឹងជាអ្វី?

យើងបាននិយាយពី "ការជូនដំណឹង" តើវាមានន័យដូចម្តេចក្នុងបរិបទ MCP?

ការជូនដំណឹងគឺជាសារដែលផ្ញើពីម៉ាស៊ីនបម្រើទៅម៉ាស៊ីនភ្ញៀវ ដើម្បី​ឲ្យដឹងអំពីជំហាន ករណី ទីតាំង ឬព្រឹត្តិការណ៍ផ្សេងៗនៅពេលការងារវែងកំពុងដំណើរការ។ ការជូនដំណឹងធ្វើឲ្យមានភាពច្បាស់លាស់ និងបង្កើនបទពិសោធន៍អ្នកប្រើ។

ឧទាហរណ៍ ម៉ាស៊ីនភ្ញៀវគួរត្រូវផ្ញើការជូនដំណឹងពេលដែលការចាប់ដៃគូដំបូងជាមួយម៉ាស៊ីនបម្រើបានបញ្ចប់។

ការជូនដំណឹងមើលទៅដូចជា​សារតាមរូបភាព JSON ៖

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ការជូនដំណឹងជារឿងមួយក្នុងប្រធានបទ MCP ត្រូវបានហៅថា ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)។

ដើម្បីអោយ logging អនុវត្តបាន ម៉ាស៊ីនបម្រើត្រូវបើកកម្មវិធីនេះជាលក្ខណៈសមត្ថភាព/មុខងារ ដូច្នេះ៖

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> អាស្រ័យលើ SDK ដែលប្រើ logging អាចត្រូវបានបើកដោយលំនាំដើម ឬអ្នកប្រហែលជាត្រូវតែបើកវាដោយផ្ទាល់ក្នុងកំណត់ค่ាម៉ាស៊ីនបម្រើរបស់អ្នក។

## ការអនុវត្តការជូនដំណឹងក្នុង MCP

ដើម្បីអនុវត្តការជូនដំណឹងនៅក្នុង MCP អ្នកត្រូវតែបង្កើតទាំងម៉ាស៊ីនបម្រើ និងម៉ាស៊ីនភ្ញៀវ ដើម្បីគ្រប់គ្រងអាប់ដេតពេលវេលាចៀសវាង។ វាធ្វើឲ្យកម្មវិធីរបស់អ្នកផ្ដល់មតិយោបល់ភ្លាមៗទៅអ្នកប្រើ ពេលបំពេញការងារវែង។

### ផ្នែកម៉ាស៊ីនបម្រើ៖ ផ្ញើការជូនដំណឹង

ចាប់ផ្តើមពីផ្នែកម៉ាស៊ីនបម្រើ។ ក្នុង MCP អ្នកកំណត់ឧបករណ៍ដែលអាចផ្ញើការជូនដំណឹង ខណៈបំពេញសំណើ។ ម៉ាស៊ីនបម្រើប្រើវត្ថុ context (ធម្មតា `ctx`) សម្រាប់ផ្ញើសារ​ទៅម៉ាស៊ីនភ្ញៀវ។

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

នៅក្នុងឧទាហរណ៍ខាងលើ ឧបករណ៍ `process_files` ផ្ញើការជូនដំណឹងបីសារទៅម៉ាស៊ីនភ្ញៀវ ពេលវាចេញបំពេញឯកសារនីមួយៗ។ វិធីសាស្រ្ត `ctx.info()` ប្រើសម្រាប់ផ្ញើសារព័ត៌មានទូទៅ។

បន្ថែមពីនេះ ដើម្បីបើកការជូនដំណឹង ត្រូវប្រាកដថាម៉ាស៊ីនបម្រើរបស់អ្នកប្រើវិធីដឹកជញ្ជូនដែលអាចផ្ទេររូបភាពបាន (ដូចជា `streamable-http`) ហើយម៉ាស៊ីនភ្ញៀវអនុវត្តកម្មវិធីគ្រប់គ្រងសារដើម្បីដំណើរការការជូនដំណឹង។ នេះជារបៀបកំណត់ម៉ាស៊ីនបម្រើប្រើ `streamable-http`៖

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]

{

    await ctx.Info("Processing file 2/3...");
```

នៅក្នុងឧទាហរណ៍ .NET នេះ ឧបករណ៍ `ProcessFiles` ត្រូវបានតាំងបរិច្ឆេទជាមួយអត្តសញ្ញាណកម្ម `Tool` ហើយផ្ញើការជូនដំណឹងបីទៅម៉ាស៊ីនភ្ញៀវ ពេលវាចេញបំពេញឯកសារ។ វិធីសាស្រ្ត `ctx.Info()` ប្រើសម្រាប់ផ្ញើសារព័ត៌មានទូទៅ។

ដើម្បីបើកការជូនដំណឹងនៅម៉ាស៊ីនបម្រើ MCP .NET របស់អ្នក ប្រសិនបើរបស់អ្នកប្រើវិធីដឹកជញ្ជូនស្ទ្រីម៖

### ផ្នែកម៉ាស៊ីនភ្ញៀវ៖ ទទួលការជូនដំណឹង

ម៉ាស៊ីនភ្ញៀវត្រូវតែអនុវត្តកម្មវិធីគ្រប់គ្រងសារដើម្បីដំណើរការនិងបង្ហាញការជូនដំណឹងពេលវាអាចមកដល់។

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

នៅក្នុងកូដខាងលើ មុខងារ `message_handler` ពិនិត្យមើលថាសារចូលមកជាជូនដំណឹងឬអត់។ ប្រសិនបើមែនវាបង្ហាញការជូនដំណឹង បើមិនមែនវាដំណើរការជាសារម៉ាស៊ីនបម្រើធម្មតា។ ក៏សូមចំណាំថា `ClientSession` ត្រូវបានបង្ហាប់ជាមួយ `message_handler` ដើម្បីគ្រប់គ្រងការជូនដំណឹងចូល។

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

នៅក្នុងឧទាហរណ៍ .NET នេះ មុខងារ `MessageHandler` ពិនិត្យថាសារចូលមកជាជូនដំណឹងឬអត់។ ប្រសិនបើមែនវាបង្ហាញការជូនដំណឹង បើមិនមែនវាដំណើរការជាសារម៉ាស៊ីនបម្រើធម្មតា។ `ClientSession` ត្រូវបានបង្ហាប់ជាមួយអ្នកដំណើរការសារតាមរយៈ `ClientSessionOptions`។

ដើម្បីបើកការជូនដំណឹង ប្រាកដថាម៉ាស៊ីនបម្រើរបស់អ្នកប្រើវិធីដឹកជញ្ជូនស្ទ្រីម (ដូចជា `streamable-http`) ហើយម៉ាស៊ីនភ្ញៀវអនុវត្តកម្មវិធីគ្រប់គ្រងសារសម្រាប់ដំណើរការការជូនដំណឹង។

## ការជូនដំណឹងជំហាន និងស្ថានភាព

ប្លុកនេះពន្យល់អំពីគំនិតនៃការជូនដំណឹងជំហាន ក្នុង MCP ហេតុអ្វីវាមានសារៈសំខាន់ និងរបៀបអនុវត្តវា ដោយប្រើ Streamable HTTP។ អ្នកនឹងបានឃើញផងដែរជម្រើសហាត់ប្រាណសម្រាប់បង្កើនការយល់ដឹងរបស់អ្នក។

ការជូនដំណឹងជំហាន គឺជាសារដែលផ្ញើពេលវេលាចៀសវាងពីម៉ាស៊ីនបម្រើទៅម៉ាស៊ីនភ្ញៀវ ខណៈកំពុងអនុវត្តប្រតិបត្តិការវែង។ មិនរង់ចាំត្រឹមបញ្ចប់វែបទាំងមូលទេ ម៉ាស៊ីនបម្រើរក្សាបើកការអាប់ដេតទៅម៉ាស៊ីនភ្ញៀវពីស្ថានភាពបច្ចុប្បន្ន។ វាបង្កើនភាពច្បាស់លាស់ បទពិសោធន៍អ្នកប្រើ និងធ្វើឲ្យការដោះស្រាយបញ្ហាស្រួលឡើង។

**ឧទាហរណ៍៖**

```text

"Processing document 2/10"
...
"Processing complete!"

```

### ហេតុអ្វីបានជាប្រើការជូនដំណឹងជំហាន?

ការជូនដំណឹងជំហានមានសារៈសំខាន់ពីហេតុផលខាងក្រោម៖

- **បទពិសោធន៍អ្នកប្រើល្អជាងមុន៖** អ្នកប្រើឃើញការអាប់ដេតពេលការងារបន្តទៅមុខ មិនមែននៅចុងក្រោយតែប៉ុណ្ណោះ។
- **មតិយោបល់ពេលវេលាចៀសវាង៖** ម៉ាស៊ីនភ្ញៀវអាចបង្ហាញរបារជំហាន ឬកំណត់ហេតុ ដែលធ្វើឲ្យកម្មវិធីមានប្រតិកម្ម។
- **ការទាន់ចិត្តប្រើប្រាស់ និងត្រួតពិនិត្យបានស្រួល៖** អ្នកអភិវឌ្ឍន៍ និងអ្នកប្រើអាចឃើញថាការប្រតិបត្តិការត្រូវបានយឺតឬឈប់កន្លែងណា។

### របៀបអនុវត្តការជូនដំណឹងជំហាន

នេះជារបៀប អ្នកអាចអនុវត្តការជូនដំណឹងជំហានក្នុង MCP៖

- **នៅលើម៉ាស៊ីនបម្រើ៖** ប្រើ `ctx.info()` ឬ `ctx.log()` ដើម្បីផ្ញើការជូនដំណឹងពេលរាល់ធាតុត្រូវបានដំណើរការ។ វាផ្ញើសារទៅម៉ាស៊ីនភ្ញៀវមុនពេលលទ្ធផលសំខាន់រួចរាល់។
- **នៅលើម៉ាស៊ីនភ្ញៀវ៖** អនុវត្តកម្មវិធីគ្រប់គ្រងសារដែលស្តាប់ និងបង្ហាញការជូនដំណឹងពេលវាគឺមកដល់។ អ្នកដំណើរការនេះបំបែកចេញពីការជូនដំណឹង និងលទ្ធផលចុងក្រោយ។

**ឧទាហរណ៍ម៉ាស៊ីនបម្រើ៖**

#### Python

```python

    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## ការពិចារណាសុវត្ថិភាព

ពេលបង្ហោះម៉ាស៊ីនបម្រើ MCP ជាមួយវិធីដឹកជញ្ជូនផ្អែកលើ HTTP សុវត្ថិភាពក្លាយទៅជាបញ្ហាសំខាន់ ដែលត្រូវការយកចិត្តទុកដាក់យ៉ាងធ្ងន់ធ្ងរ​ពីចំណុចប្រហារច្រើន និងយន្តការពារ។

### សង្ខេប

សុវត្ថិភាពពិតជារំខានពេលបើកម៉ាស៊ីនបម្រើ MCP តាម HTTP។ Streamable HTTP បង្កើតផ្ទៃដីស្មើនិងតម្រូវឲ្យមានការកំណត់រចនាសម្ព័ន្ធយ៉ាងម៉ត់ចត់។

### ចំណុចសំខាន់

- **ការត្រួតពិនិត្យហេដ្ឋារចនាសម្ព័ន្ធ Origin**៖ តែងតែត្រួតពិនិត្យប<a>្រាំង Origin ដើម្បីការពារ DNS rebinding។
- **ការចងភ្ជាប់ជាមួយ localhost**៖ សម្រាប់អភិវឌ្ឍន៍ក្នុងតំបន់ ចងម៉ាស៊ីនបម្រើទៅ `localhost` ដើម្បីមិនបង្ហាញទៅអ៊ីនធឺណែតសាធារណៈ។
- **ការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ**៖ អនុវត្តការផ្ទៀងផ្ទាត់ (ឧ. ប្រើ API keys, OAuth) សម្រាប់ការបញ្ចេញផលិតកម្ម។
- **CORS**៖ កំណត់គោលការណ៍ Cross-Origin Resource Sharing ដើម្បីហាមការចូលប្រើ។
- **HTTPS**៖ ប្រើ HTTPS នៅក្នុងបរិបទផលិតកម្ម ដើម្បីបំលែងចរាចរណ៍។

### លំហូរ​បញ្ហា

- ការប្រកួតប្រជែងរវាងសុវត្ថិភាព និងភាពងាយស្រួលក្នុងការអភិវឌ្ឍ
- ធានាការភ្ជាប់ការងារជាមួយបរិយាកាសម៉ាស៊ីនភ្ញៀវនានា

## ការវិវត្តពី SSE ទៅ Streamable HTTP
សម្រាប់កម្មវិធីដែលកំពុងប្រើ Server-Sent Events (SSE) នៅពេលនេះ ការផ្លាស់ប្តូរទៅ Streamable HTTP អាចផ្តល់នូវសមត្ថភាពកាន់តែល្អ និងជួយធានាការរស់រានមាននៅរយៈពេលវែងសម្រាប់ការអនុវត្ត MCP របស់អ្នក។

### ហេតុផលដែលត្រូវធ្វើបច្ចុប្បន្នភាព?

មានហេតុផលសង្កត់សម្រាប់ធ្វើបច្ចុប្បន្នភាពពី SSE ទៅ Streamable HTTP ៖

- Streamable HTTP ផ្តល់នូវសមត្ថភាពអាចពង្រីកបានកាន់តែល្អ ការប្រកាប់គ្នាល្អ និងគាំទ្រជូនការជូនដំណឹងមានទ្រង់ទ្រាយច្រើនជាង SSE។
- វាផ្តល់ជាមធ្យោបាយដឹកជញ្ជូនដែលបានផ្តល់អនុសាសន៍សម្រាប់កម្មវិធី MCP ថ្មីៗ។

### ជំហានផ្លាស់ប្តូរ

នេះជាវិធីដែលអ្នកអាចផ្លាស់ប្តូរពី SSE ទៅ Streamable HTTP ក្នុងកម្មវិធី MCP របស់អ្នក៖

- **បន្ទាន់សម័យកូដម៉ាស៊ីនបម្រើ** ដើម្បីប្រើ `transport="streamable-http"` នៅក្នុង `mcp.run()`។
- **បន្ទាន់សម័យកូដអតិថិជន** ដើម្បីប្រើ `streamablehttp_client` ជំនួស SSE client។
- **អនុវត្ត​គ្រប់គ្រងសារមួយ** នៅចំណុចអតិថិជនសម្រាប់ដំណើរការជូនដំណឹង។
- **សាកល្បងសម្របសម្រួល** ជាមួយឧបករណ៍ និងដំណើរការដែលមានស្រាប់។

### ការរក្សា Compatibility

វាត្រូវបានផ្តល់អនុស្សាធិថគប្បីរក្សា compatibility ជាមួយអតិថិជន SSE មានស្រាប់ក្នុងអំឡុងពេលផ្លាស់ប្តូរ។ នេះជាគោលការណ៍ខ្លះៗ ៖

- អ្នកអាចគាំទ្រទាំងពី SSE និង Streamable HTTP ដោយរត់គ្រប់មធ្យោបាយពីរនៅលើចំណុចបញ្ចូលខុសៗគ្នា។
- ធ្វើការផ្លាស់ប្តូរអតិថិជនទៅកាន់មធ្យោបាយថ្មីយ៉ាងយឺតយ៉ាវ។

### បញ្ហាប្រឈម

ត្រូវប្រាកដថាអ្នករៀបចំដោះស្រាយបញ្ហាប្រឈមខាងក្រោមក្នុងអំឡុងពេលផ្លាស់ប្តូរ៖

- ប្រាកដថាអតិថិជនទាំងអស់បានបន្ទាន់សម័យ
- ដោះស្រាយភាពខុសគ្នានៅក្នុងការផ្តល់ជូនដំណឹង

## ការពិចារណាអំពីសុវត្ថិភាព

សុវត្ថិភាពគួរតែជាការប្តេជ្ញាធ្វើជាអាទិភាពខ្ពស់នៅពេលអនុវត្តម៉ាស៊ីនបម្រើណាមួយ ជាពិសេសពេលប្រើប្រាស់ការដឹកជញ្ជូនមូលដ្ឋាន HTTP ដូចជា Streamable HTTP ក្នុង MCP។

នៅពេលអនុវត្តម៉ាស៊ីនបម្រើ MCP ជាមួយការដឹកជញ្ជូនមូលដ្ឋាន HTTP សុវត្ថិភាពក្លាយជាបញ្ហាដ៏សំខាន់ដែលត្រូវមានការផ្តោតអារម្មណ៍យ៉ាងម៉ត់ចត់លើចំណុចវាយប្រហារ និងយន្តការការពារ។ 

### ទិដ្ឋភាពទូទៅ

សុវត្ថិភាពមានសារៈសំខាន់ខ្លាំងនៅពេលបង្ហាញម៉ាស៊ីនបម្រើ MCP តាម HTTP។ Streamable HTTP បន្ថែមចំណុចវាយប្រហារថ្មី និងតម្រូវឲ្យមានការកំណត់អោយត្រឹមត្រូវយ៉ាងម៉ត់ចត់។

នេះជាចំណុចសុវត្ថិភាពសំខាន់ៗ៖

- **ការValidate Origin Header**៖ ត្រូវតែត្រួតពិនិត្យ `Origin` header ជានិច្ច ដើម្បីការពារការវាយប្រហារ DNS rebinding។
- **Localhost Binding**៖ សម្រាប់ការអភិវឌ្ឍន៍ក្នុងតំបន់​ជាប់កន្លែងស្នាក់នៅ រួចភ្ជាប់ម៉ាស៊ឺរទៅ `localhost` ដើម្បីមិនឲ្យបង្ហាញទៅអ៊ីនធឺណិតសាធារណៈ។
- **សម្គាល់អត្តសញ្ញាណ**៖ អនុវត្តសម្គាល់អត្តសញ្ញាណ (ឧ. កូនសោ API, OAuth) សម្រាប់ការបញ្ជូនផលិតកម្ម។
- **CORS**៖ កំណត់គោលការណ៍ការចែករំលែកតំបន់ដើម (CORS) ដើម្បីកំណត់ការចូលប្រើ។
- **HTTPS**៖ ប្រើ HTTPS ក្នុងផលិតកម្មសម្រាប់ប៊ិចទិន្នន័យ។

### អនុវត្តន៍ល្អបំផុត

បន្ថែមទៀត អ្នកគួរតែអនុវត្តន៍អនុវត្តន៍ល្អខាងក្រោម នៅពេលបង្កើតម៉ាស៊ីនបម្រើ MCP ស្ទ្រីមម ing ដែលមានសុវត្ថិភាព៖

- មិនគួរជឿទុកចិត្តសំណើដែលមិនបានត្រួតពិនិត្យ។
- កត់ត្រា និងត្រួតពិនិត្យការចូលប្រើ និងកំហុសទាំងអស់។
- ធ្វើបច្ចុប្បន្នភាពខ្នាតបានជាប្រចាំសម្រាប់ការជួសជុលវីរុសសុវត្ថិភាព។

### បញ្ហាប្រឈម

អ្នកនឹងប្រឈមមុខនឹងបញ្ហាប្រឈមខ្លះៗនៅពេលអនុវត្តសុវត្ថិភាពរបស់ម៉ាស៊ីនបម្រើ MCP streaming៖

- ការប្រកបដោយតុល្យភាពរវាងសុវត្ថិភាព និងភាពងាយស្រួលដល់ការអភិវឌ្ឍន៍
- ធានា Compatibility ជាមួយបរិយាកាសអតិថិជនផ្សេងៗគ្នា

### កិច្ចការចាត់ចែង៖ បង្កើតកម្មវិធី MCP Streaming ផ្ទាល់ខ្លួន

**សេណារីយ៉ូ៖**  
បង្កើតម៉ាស៊ីនបម្រើ MCP និងអតិថិជន ដែលម៉ាស៊ីនបម្រើដំណើរការបញ្ជីធាតុ (ឧ. ឯកសារ ឬឯកសារសារ) ហើយផ្ញើជូនដំណឹងសម្រាប់ធាតុមួយៗដែលបានដំណើរការ។ អតិថិជនគួរតែបង្ហាញការជូនដំណឹងនីមួយៗនៅពេលវាផ្ដល់មក។

**ជំហាន៖**

1. អនុវត្តឧបករណ៍ម៉ាស៊ីនបម្រើដែលដំណើរការបញ្ជីហើយផ្ញើការជូនដំណឹងសម្រាប់ធាតុមួយៗ។
2. អនុវត្តអតិថិជនជាមួយគ្រប់គ្រងសារដើម្បីបង្ហាញការជូនដំណឹងពេលពិត។
3. សាកល្បងការអនុវត្តទាំងពីរ ដោយរត់ម៉ាស៊ីនបម្រើ និងអតិថិជន ហើយមើលការជូនដំណឹង។
